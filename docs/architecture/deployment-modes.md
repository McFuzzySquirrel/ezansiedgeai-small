# Deployment Modes — eZansiEdgeAI

## Overview

eZansiEdgeAI supports three deployment modes. Each mode is a strict superset
of the previous one — capabilities are additive, never substitutive. The
phone-only mode is the baseline that must always work.

---

## Mode A — Phone Only

**Scenario:** Learner at home, no WiFi, no school device, no internet.
This is the worst-case deployment and the most common real-world scenario.

### What the learner gets

| Capability | Available | Notes |
|------------|-----------|-------|
| Text chat tutoring | ✅ | Full retrieval + explanation pipeline |
| Content browsing | ✅ | Installed packs only |
| Worked examples | ✅ | Generated by local LLM |
| Learner preferences | ✅ | Stored locally |
| Voice input | ❌ | No local STT in V1 |
| Voice output | ❌ | No local TTS in V1 |
| Content updates | ❌ | Must connect to edge or internet later |
| Complex query offload | ❌ | Phone model only |

### Network topology

```
┌──────────────┐
│  Learner     │
│  Phone       │    (standalone, no network)
│              │
│  App + AI +  │
│  Data layers │
│  all local   │
└──────────────┘
```

### Technical requirements

- Pre-installed content pack(s) — sideloaded via USB, Bluetooth share,
  or initial download.
- Quantized LLM + embedding model on device.
- No runtime network dependency.

### Fallback behaviour

This IS the fallback. Every other mode degrades gracefully to Mode A.

## Mode B — Phone + School Edge WiFi

**Scenario:** Learner at school, connected to the school's local WiFi network
where an edge device is running. This is the **primary target deployment**.

### What the learner gets (above Mode A)

| Capability | Available | Notes |
|------------|-----------|-------|
| All Mode A features | ✅ | Baseline preserved |
| Voice input (STT) | ✅ | Via edge Whisper instance |
| Voice output (TTS) | ✅ | Via edge Piper instance |
| Content pack updates | ✅ | LAN sync from edge cache |
| Delta pack updates | ✅ | Bandwidth-efficient incremental updates |
| Faster pack download | ✅ | LAN speed vs. mobile data |
| Complex query offload | ⏳ V2 | Larger model on edge for hard questions |

### Network topology

```
    School WiFi Router
         │
    ┌────┴──────────────┐
    │                   │
┌───▼──────┐    ┌───────▼──────┐
│ Edge     │    │ Learner      │
│ Device   │◄──►│ Phone(s)     │
│          │    │              │
│ Content  │    │ Discovers    │
│ STT/TTS  │    │ edge via     │
│ Updates  │    │ mDNS         │
└──────────┘    └──────────────┘
        (10–40 phones per classroom)
```

### Connection lifecycle

1. **Discovery:** Phone app starts mDNS query for `_ezansi._tcp.local`
   on network join.
2. **Capability check:** Phone calls `GET /capabilities` on discovered
   edge device.
3. **UI adaptation:** Voice button appears if STT is available. Content
   update badge appears if newer packs exist.
4. **Session:** Phone uses edge services opportunistically during the
   school session.
5. **Disconnection:** When phone leaves WiFi, all edge-dependent UI
   elements disappear silently. Active text conversations continue
   uninterrupted.

### Fallback behaviour

| Event | Behaviour |
|-------|-----------|
| Edge device powers off (load shedding) | Phone continues with text-only; voice buttons disappear |
| WiFi drops temporarily | Phone retries discovery every 30 s; continues locally |
| Edge STT queue full | Phone shows "voice busy, try again" — text still works |
| Edge returns error | Phone logs error locally; falls back to local pipeline |

**Invariant:** No user-visible error dialog. Degradation is silent and
graceful.

## Mode C — Community Hub Device

**Scenario:** A shared device at a library, NGO, community centre, or
teacher's home. Multiple learners connect their phones to the hub's WiFi.

### Differences from Mode B

| Aspect | Mode B (School) | Mode C (Community Hub) |
|--------|-----------------|----------------------|
| Location | School premises | Library, NGO, community centre |
| WiFi | School's existing network | Hub creates its own AP |
| Uptime | School hours | Flexible / scheduled hours |
| Admin | Teacher or IT contact | Volunteer or librarian |
| Internet | Rare | Occasionally available |
| Learner count | 20–40 per class | 5–15 per session |

### Network topology

```
┌──────────────────────────────────────────┐
│           Community Hub Device            │
│                                          │
│  ┌────────────┐  ┌────────────────────┐  │
│  │ WiFi AP    │  │ Edge Services      │  │
│  │ (hostapd)  │  │ (same as Mode B)   │  │
│  └─────┬──────┘  └────────────────────┘  │
│        │                                 │
└────────┼─────────────────────────────────┘
         │
    ┌────┴──────────────┐
    │                   │
┌───▼──────┐    ┌───────▼──────┐
│ Learner  │    │ Learner      │
│ Phone 1  │    │ Phone 2...N  │
└──────────┘    └──────────────┘
```

### Key difference: Hub IS the WiFi

In Mode C the hub device runs its own wireless access point (hostapd + dnsmasq).
Phones connect directly to the hub's WiFi. This means:

- No dependency on existing school infrastructure.
- Hub controls the network (DHCP, DNS, mDNS all local).
- Can work in locations with no existing WiFi.
- Hub can be battery-backed (USB power bank) for load shedding resilience.

### Setup

```bash
sudo ./ezansi-setup.sh --mode hub \
    --ssid "eZansi-Library" \
    --channel 6 \
    --school-name "Thabo Mbeki Library"
```

The setup script configures:
1. hostapd for WiFi AP creation.
2. dnsmasq for DHCP + local DNS.
3. All edge services (same as Mode B).
4. Captive portal redirect to content library (optional).

## Content Pack Distribution by Mode

| Mode | How packs arrive |
|------|-----------------|
| A | Pre-installed at app install, sideloaded via USB/Bluetooth, or downloaded when internet is briefly available |
| B | Synced from edge device over LAN (`GET /content/packs/{id}`). Edge is updated by teacher when internet is available |
| C | Same as B, but hub may be pre-loaded with packs via USB before deployment |

Pack update flow:

```
Internet ──(opportunistic)──► Edge/Hub ──(LAN)──► Phone(s)
                                │
                          USB sideload
                          (offline path)
```

## Deployment Decision Guide

| Question | Recommendation |
|----------|---------------|
| School has no WiFi, no shared device? | Mode A — phone only |
| School has WiFi and can host a small device? | Mode B — primary target |
| Community space wants to serve multiple learners? | Mode C — hub with own AP |
| Teacher wants to serve learners at home? | Mode C on a laptop |
| NGO distributing devices? | Pre-configure as Mode C hubs |

---

## Load Shedding & Power Failure Matrix

| Event | Mode A | Mode B | Mode C |
|-------|--------|--------|--------|
| Power out at school | No impact (phone on battery) | Phone continues (Mode A fallback). Edge offline. | Hub offline; phones continue standalone. |
| Phone battery dies | Learning stops for that learner | Learning stops for that learner | Learning stops for that learner |
| Edge/Hub restarts | N/A | Phone re-discovers automatically | Phone re-discovers automatically |
| WiFi router dies | N/A | Falls back to Mode A | Hub IS the AP; N/A unless hub dies |
