# Decisions Made
- Decision: Use llama-cpp-python for benchmarking (not ONNX Runtime)
  - Reason: llama.cpp is the primary candidate runtime for production (per phone-architecture.md). Benchmarking with the same runtime gives directly applicable results. GGUF is the most widely available quantised format.
  - Impact: Models must be GGUF format; ONNX comparison deferred unless llama.cpp results are marginal.

- Decision: Benchmark on host machine first, not Android emulator
  - Reason: Validates model quality and relative performance quickly. Android emulator benchmarking is a follow-up step once a model shortlist is established.
  - Impact: Absolute latency numbers won't match phone hardware — but relative rankings and quality assessments are valid.

- Decision: Adopt Qwen2.5-1.5B-Instruct (Q4_K_M GGUF) as V1 on-device LLM (ADR 0006)
  - Reason: Only model passing all 3 P0-001 acceptance criteria (load < 5s, gen < 10s, RAM < 2 GB). Benchmarked across 12 prompts × 3 runs with empirical evidence.
  - Impact: Constrains AI layer, prompt engineering, content pack format, storage budget. SmolLM2 retained as backup for 4 GB+ devices.

# Key Learnings
- Model size on disk ≠ RAM usage. Qwen2.5 (1066 MB on disk) uses 1839 MB RAM; SmolLM2 (1007 MB on disk) uses 2214 MB RAM — SmolLM2 has a higher RAM-to-disk ratio.
- Qwen2.5-1.5B produces significantly more tokens (150 consistently) but takes longer per token (~19 tok/s vs SmolLM2's variable output). SmolLM2 often generates far fewer tokens (short responses), which inflates its speed metrics.
- The 2 GB RAM budget is extremely tight. Only 1 of 4 candidates fits, and it uses 90% of budget (1839/2048 MB). This leaves minimal headroom for the embedding model, vector DB, and app overhead.
- llama-cpp-python builds and runs smoothly on Linux x86_64 — no runtime issues.
- Host benchmarks give useful relative rankings but absolute RAM numbers may differ on Android (mmap behaviour, OS memory accounting).

# If Repeating This Work
Do this: Start with smallest models first (saves download time); run a 3-prompt quick validation before the full 12×3 suite; check if n_ctx can be reduced to save RAM
Avoid this: Don't skip the warmup run — first inference is always slower; don't compare tok/s across models with different output lengths
Watch out for: SmolLM2 generates very short responses despite max_tokens=150 — this masks true latency; RAM measurements include Python process overhead (~50-100 MB) that won't exist in a native Android app

# Future Agent Guidance
Prefer: Running benchmarks with reproducible seeds; using the same prompt template as production
Avoid: Making final model decisions from host-machine benchmarks alone; testing only English prompts
