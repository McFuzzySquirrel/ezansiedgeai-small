# Decisions Made
- Decision: Security/privacy handled as constraints across all agents, not a separate agent
  - Reason: Security requirements (§10) are constraints on every agent's domain—encryption in learner-data, permissions in project-architect, network isolation in all agents. A separate security agent would overlap with every other agent.
  - Impact: Each agent is responsible for security within its domain; qa-test-engineer owns security audit/verification.

- Decision: 8 agents (7 specialist + 1 orchestrator) for team size
  - Reason: Project has 10 distinct functional areas, but some naturally combine (profiles+preferences+persistence → learner-data-engineer). 8 agents provides full coverage without unnecessary granularity.
  - Impact: Every PRD requirement maps to exactly one agent; no gaps or overlaps.

- Decision: Two reusable skills — scaffold-android-feature and create-content-chunk
  - Reason: Feature modules (5 screens) all follow the same structure, and content chunks (~100+ files) all follow the same format. Both are high-frequency repeatable patterns.
  - Impact: Consistent structure across all feature modules and content files.

# Key Learnings
Technical insights:
- The PRD's 4-layer architecture (App→AI→Data→Hardware) maps cleanly to agent boundaries
- Sequential model loading (embedding before LLM) is a critical cross-agent concern between ai-pipeline-engineer and project-architect
