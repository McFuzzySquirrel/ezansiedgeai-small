# Experiments / Evidence
- RAM headroom calculation: 4 GB marketed → ~3 GB usable → ~3,000 MB - 1,839 MB (Qwen2.5) = ~1,161 MB for embedding model + vector DB + app
- This is a 5.5× improvement over the previous 209 MB headroom
- **P0-002 benchmark results (12 combinations):**
  - all-minilm-l6-v2: 100% retrieval accuracy, ~10ms query embed, 87.3 MB model, ~554 MB peak RAM — **PASS all 4 stores**
  - bge-small-en-v1.5: 95% accuracy, ~18ms embed, 128.1 MB model (exceeds 100 MB limit) — **FAIL all 4 stores**
  - gte-small: 95% accuracy, ~30ms embed, 64.4 MB model, ~553 MB peak RAM — **PASS all 4 stores**
  - Vector store choice has negligible impact: all stores <0.1ms search, <150 KB index at 50 chunks
  - FAISS IVF warning: only 50 training points for 8 centroids (needs ≥312) — irrelevant at content-pack scale
- **Winner:** all-minilm-l6-v2 — best accuracy (100%), fastest embed (~10ms), fits size limit (87 MB < 100 MB)
- **Storage footprint validated:** APK 50 MB + LLM 1,066 MB + embedding 87 MB + content 200 MB = ~1,403 MB first-launch download

# Decisions Made
- Decision: Raise minimum device RAM from 3 GB to 4 GB (marketed) / ~3 GB usable
  - Reason: 209 MB headroom with 3 GB devices is too tight for embedding model + vector DB + app. 4 GB gives ~1,161 MB — sufficient margin.
  - Impact: Drops Samsung A04 2 GB SKU from target devices. A04s/A05 (4 GB), Redmi 10C (4 GB), Mobicel Hero (4 GB) remain valid.

- Decision: Fold P0-003 storage scope into P0-002
  - Reason: Storage budget (150 MB) was already known to be unrealistic after P0-001 (Qwen2.5 = 1,066 MB). Measuring embedding model + vector index storage naturally belongs alongside the embedding spike.
  - Impact: P0-002 produces a single ADR covering embedding model, vector store, storage budget, and RAM floor. P0-003 becomes purely battery & thermal.

- Decision: Rename P0-003 to Battery & Thermal Validation
  - Reason: Storage scope moved to P0-002. P0-303 in Phase 3 already covered battery & thermal but only post-Phase 2. Having a Phase 0 item validates thermal constraints early.
  - Impact: Clearer backlog item naming. No scope gap.

- Decision: all-MiniLM-L6-v2 is the recommended embedding model (pending ADR 0007)
  - Reason: Only model achieving 100% retrieval accuracy. Fastest embedding (10ms vs 18ms BGE, 30ms GTE). Fits size limit (87 MB < 100 MB). Well within RAM budget (554 MB peak vs 1,161 MB headroom).
  - Alternatives: bge-small-en-v1.5 (FAIL — 128 MB exceeds 100 MB limit), gte-small (PASS — smaller at 64 MB but 3× slower and 5% lower accuracy)
  - Impact: Unblocks P0-004 (content pack with pre-computed embeddings) and P0-005 (E2E pipeline).

- Decision: Vector store choice is deferred — all stores equivalent at content-pack scale
  - Reason: All 4 stores show <0.1ms search time and identical retrieval results at 50 chunks. Difference only matters at thousands of chunks.
  - Alternatives: FAISS Flat (simplest, exact), FAISS IVF (ANN but needs more training data), HNSWlib (graph ANN), NumPy cosine (zero-dependency)
  - Impact: For P0-004, use FAISS Flat (simplest). Revisit if content scales beyond ~1,000 chunks.

# Key Learnings
Technical insights:
- 4 GB devices are mainstream in the SA low-end market (A04s, A05, Redmi 10C, Mobicel Hero all have 4 GB variants)
- RAM headroom relief from 209 MB to ~1,161 MB is significant — removes RAM as a project-killer risk for embedding model selection
- all-MiniLM-L6-v2 outperforms on accuracy (100%) despite being mid-sized (87 MB) — model size ≠ quality for this domain
- Vector store choice is irrelevant at content-pack scale (50–500 chunks) — all stores <0.1ms
- Embedding model RAM overhead is tiny (~29 MB load delta) — the bottleneck is the LLM, not the embedding model
- PyTorch CPU-only build (~200 MB) is sufficient for embedding inference — no need for CUDA on mobile
- python3-dev headers required for hnswlib compilation — note for CI/CD setup

# If Repeating This Work
Do this: Raise device floor early when a single component (LLM) consumes >90% of the budget
Avoid this: Tiered device strategies add complexity — prefer raising the floor if the market supports it
Watch out for: Confirm actual device availability in target market before raising specs

# Future Agent Guidance
Prefer: Simple constraint changes over complex architectural workarounds
Avoid: Introducing tiered/conditional logic when a floor change suffices
