# Decisions Made
- Decision: Exclude `*.task` files via `models/.gitignore` and untrack `gemma4-1b.task`.
  - Reason: MediaPipe `.task` bundles are large binary model files (529 MB) that don't belong in version control. The existing gitignore covered other formats (`*.gguf`, `*.onnx`) but missed `*.task` — a gap introduced when Gemma 4 replaced Qwen2.5 as the primary LLM.
  - Impact: Model binaries will no longer be accidentally committed. Contributors are directed to Kaggle for download and told to use `adb push` for device deployment. This keeps the repo lean and the workflow explicit.

- Decision: Document Gemma 4 as the primary model in `models/phone-models/README.md`, with Qwen2.5 as a superseded legacy section.
  - Reason: The README still described Qwen2.5 as the primary model despite ADR 0012 selecting Gemma 4. Keeping Qwen2.5 as a reference helps with fallback testing.
  - Impact: Future contributors get accurate, actionable instructions without confusion about which model is current.

# Key Learnings
Technical insights:
- When adding a new model format (e.g., `.task` for MediaPipe), always audit the gitignore in the same PR/commit — format gaps cause binaries to slip through.
- `git rm --cached <file>` safely removes a binary from the index without deleting it locally. Essential when a large file has already been staged or committed.

Prompting insights:
- "Make sure we're not pushing model files" → agent should immediately check both `.gitignore` rules and `git ls-files` (not just assume gitignore is correct).

Tooling insights:
- `git ls-files models/` is the most reliable check for what's actually tracked — not `git status` alone.

# If Repeating This Work
Do this: Run `git ls-files models/` as the first check before any commit involving model files.
Avoid this: Assuming a pre-existing `.gitignore` covers all new file extensions — verify explicitly when a new model format is introduced.
Watch out for: `git status` showing `??` for untracked files while also having previously-committed binaries silently staged. Always `git ls-files` to confirm.

# Future Agent Guidance
Prefer: Auditing `git ls-files <directory>` before any pre-push check — faster and more reliable than scanning gitignore rules manually.
Avoid: Editing gitignore without also checking whether the binary is already tracked (`git ls-files`). The gitignore only prevents *new* additions — already-tracked files need `git rm --cached`.
