# Decisions Made
1. **Content pack format: SQLite (.pack)**
   - Decision: Use SQLite as the pack file format with 4-table schema (manifest, chunks, embeddings, faiss_indexes)
   - Reason: Native Android support (no extra libraries), queryable without extraction, transactional writes, standard tooling
   - Alternatives considered: ZIP/compressed directory (simpler, easier to inspect but extraction required), custom binary (rejected early — no parser, no tooling)
   - Impact: P0-103 (Content Pack Loader) uses Android SQLiteDatabase API directly. P0-104 (ExplanationEngine) loads FAISS index from BLOB via temp file.
   - ADR: 0008

2. **Chunk sourcing: hybrid (P0-002 existing + CAPS gaps)**
   - Decision: Refine 10 of the 15 existing fractions chunks from the P0-002 spike dataset; no gaps needed (all CAPS Term 1 topics covered by the 15)
   - Reason: Faster than authoring from scratch; existing chunks proven at 100% retrieval accuracy in P0-002
   - Alternatives: Pure reuse (all 15 — exceeded 10-chunk target), fresh authoring (slower, no proven accuracy data)
   - Impact: Pack has verified accuracy baseline from day 1

3. **P0-005 dev-machine latency threshold: 60s (not 15s)**
   - Decision: Use 60s as the `total_latency_s` acceptance criterion in config.yaml for the dev-machine smoke test
   - Reason: Dev machine's Quadro M2200 GPU (sm_52) is incompatible with PyTorch 2.10+ (requires sm_70+); all inference runs CPU-only at ~17-19s/question vs the 15s on-device target
   - Alternatives: Install CPU-only PyTorch build (would require separate venv configuration), measure 15s on Android emulator (not available in this session)
   - Impact: On-device 15s target still valid; p0-005 proves architecture correctness, not absolute on-device performance. Latency profiling on real hardware in P0-104.

# Key Learnings
- SQLite is the right format for Android content packs — no extraction, native API, per-row integrity checks
- The P0-002 spike artefacts (chunks.json, test_queries.json, local models, venv) are directly reusable for P0-004 — no re-download or re-author needed
- 76 KB pack for 10 chunks × 384-dim embeddings + FAISS index — far below the 200 MB budget
- The `build_pack.py` + `validate_pack.py` pattern (build → validate → exit code) is good for CI integration in Phase 2
- P0-005: Sequential model loading (embed model unloads before LLM loads) is confirmed working — embedding frees 49 MB before LLM takes +1754 MB. ADR 0007 constraint satisfied.
- P0-005: SentenceTransformer requires explicit `device="cpu"` on machines where PyTorch CUDA initialisation fails (sm_52 GPU incompatibility). Without it, the import succeeds but encode() crashes fatally.
- P0-005: The RAG loop works: retrieved chunks are semantically correct, and generated answers reference chunk content with no hallucination observed in 5 test questions.

# If Repeating This Work
Do this: reuse existing venv and locally downloaded models; chunk selection by CAPS term + topic scope works well; use `device="cpu"` for SentenceTransformer to avoid GPU-compatibility issues during development  
Avoid this: authoring chunks from scratch when spike data already exists and has proven accuracy; assuming GPU is available on dev machine  
Watch out for: FAISS index requires a temp-file round-trip to load (C++ API limitation) — this is expected, not a bug; llama-cpp-python pip install can produce large log output and silently skip on first `pip install -r requirements.txt` — install explicitly if import fails

# Future Agent Guidance
Prefer: SQLite for any structured binary artifact that needs to be read on Android without extraction  
Avoid: storing large binary blobs (models) in the .pack file — models are downloaded separately per ADR 0007  
Always: use `device="cpu"` when initialising SentenceTransformer in dev environments where PyTorch GPU compatibility is not verified  
Always: use config-driven thresholds in pipeline runners — don't hardcode numeric targets in report writer/print functions
