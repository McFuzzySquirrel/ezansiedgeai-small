Research and document findings on two identified gaps: (1) the current Qwen2.5-1.5B GGUF model is slow for on-device inference, and (2) the app lacks user-facing semantic search capability. Evaluate Google's Gemma 4 model family as a potential replacement and define what's needed to enable semantic search.

Key findings:

2. Gemma 4 1B (INT4) could replace both Qwen2.5-1.5B + all-MiniLM-L6-v2 with a single ~600 MB model, potentially halving disk usage, RAM, and generation latency.

4. MediaPipe/LiteRT provides first-party GPU-accelerated inference for Gemma on Android, eliminating the llama.cpp JNI complexity.

# Key Learnings

- **Technical:** The OnnxEmbeddingModel's real inference is not yet implemented — it always falls back to hash-based embedding. This means the current RAG pipeline's retrieval quality depends entirely on pre-computed embeddings in content packs, not runtime query embedding quality. Semantic search cannot work until real embedding inference is integrated.
- **Architecture:** The interface abstraction (LlmEngine, EmbeddingModel, ContentRetriever) makes model swaps clean. Adding GemmaLiteRtEngine and GemmaEmbeddingModel alongside existing implementations is straightforward.
- **Research:** The Gemma 4 blog post URL was inaccessible from the sandbox environment. Future research sessions should pre-fetch external content or have the human provide key data points.
