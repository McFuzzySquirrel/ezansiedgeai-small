# ADR 0006 — Qwen2.5-1.5B as on-device LLM

## Session Journey

[ejs-session-2026-03-03-02](../journey/2026/ejs-session-2026-03-03-02.md)

## Context

The eZansiEdgeAI platform requires a quantized small language model (SLM) to run entirely on-device on low-end Android phones (3 GB total RAM, 2 GB app budget). The model must generate coherent, curriculum-aligned explanations for Grade 6 Mathematics (CAPS) within strict latency and memory budgets defined in the V1 backlog (P0-001):

- Model loads in < 5 seconds
- Generates a coherent 150-token response in < 10 seconds
- Peak RAM usage stays below 2 GB

The phone architecture (ADR 0001) mandates phone-first design. The offline-first design (ADR 0002) mandates zero network calls. The retrieval-first approach (ADR 0003) means the model explains retrieved content — it does not invent content. These constraints narrow the field to quantized models in the 1–2 GB range using GGUF format with llama.cpp as the inference runtime.

## Session Intent

Execute the P0-001 feasibility spike: benchmark candidate quantized SLMs against the acceptance criteria to determine if on-device LLM inference is viable, and if so, which model to use.

## Collaboration Summary

The agent scaffolded a complete benchmark harness (download scripts, benchmark runner, report generator) with 12 CAPS-aligned Grade 6 Maths prompts. All 4 candidate models were downloaded from HuggingFace and benchmarked across 12 prompts × 3 runs each. Results were automatically collated into a spike report. The human reviewed the rubric and confirmed an ADR was warranted.

## Decision Trigger / Significance

Choosing the on-device LLM is one of the most consequential implementation decisions in the project. It:

- **Sets a system boundary** — the entire AI layer (prompt template engine, context window sizing, output post-processing) is designed around this model's capabilities and constraints.
- **Creates a public contract** — content packs must be authored within this model's context window and instruction-following characteristics. Changing the model later forces re-validation of all packs.
- **Has long-lived consequences** — the storage budget (P0-003), embedding pipeline (P0-002), and end-to-end pipeline (P0-005) all depend on this choice. Reversing it touches every layer of the stack.
- **Is backed by empirical evidence** — 4 models benchmarked with quantified results, not a theoretical selection.

## Considered Options

### Option A: Qwen2.5-1.5B-Instruct (Q4_K_M GGUF)

- **Size:** 1,066 MB on disk
- **Peak RAM:** 1,839 MB (90% of 2 GB budget)
- **Load time:** 0.784s
- **Avg generation (150 tokens):** 8.0s
- **Throughput:** 19.0 tok/s
- **Strengths:** Only model passing all 3 acceptance criteria. Good multilingual support (relevant for Afrikaans V1, African languages V2). Consistently generates full 150-token responses. Highest throughput of all candidates.
- **Weakness:** Tight RAM headroom (~209 MB left for embedding model, vector DB, app overhead).

### Option B: SmolLM2-1.7B-Instruct (Q4_K_M GGUF)

- **Size:** 1,007 MB on disk
- **Peak RAM:** 2,214 MB (exceeds 2 GB budget by 166 MB / 8%)
- **Load time:** 0.849s
- **Avg generation:** 0.895s (fastest by far)
- **Throughput:** 10.2 tok/s
- **Strengths:** Extremely fast generation. Smallest disk footprint.
- **Weakness:** Exceeds RAM budget. Generates very short responses (often well below 150 tokens), which inflates speed metrics. Higher RAM-to-disk ratio than Qwen2.5 (2.2× vs 1.7×).

### Option C: Gemma-2-2B-it (Q4_K_M GGUF)

- **Size:** 1,629 MB on disk
- **Peak RAM:** 2,900 MB (exceeds budget by 852 MB / 42%)
- **Load time:** 1.247s
- **Avg generation:** 12.5s (exceeds 10s limit)
- **Strengths:** Good reasoning quality from Google's instruction tuning.
- **Weakness:** Fails on both generation time and RAM. Too large for target devices.

### Option D: Phi-3.5-mini-instruct (Q4_K_M GGUF)

- **Size:** 2,282 MB on disk
- **Peak RAM:** 4,379 MB (exceeds budget by 2,331 MB / 114%)
- **Load time:** 1.420s
- **Avg generation:** 17.6s (exceeds 10s limit by 76%)
- **Strengths:** Strongest reasoning capability of all candidates.
- **Weakness:** Far too large for target hardware. Fails decisively on both generation time and RAM. 3.8B parameters is too many for a 3 GB device.

---

# Decision

**Adopt Qwen2.5-1.5B-Instruct (Q4_K_M GGUF) as the V1 on-device LLM, with llama.cpp as the inference runtime.**

SmolLM2-1.7B-Instruct is retained as a backup candidate if the RAM budget can be relaxed (e.g. on 4 GB devices, or with more aggressive quantisation).

---

# Rationale

Qwen2.5-1.5B is the only model that passes all three acceptance criteria from the P0-001 spike:

| Criterion | Target | Qwen2.5-1.5B | Next Best (SmolLM2) |
|-----------|--------|-------------|---------------------|
| Load time | < 5s | 0.784s ✅ | 0.849s ✅ |
| 150-token generation | < 10s | 8.0s ✅ | 0.895s ✅ (but short outputs) |
| Peak RAM | < 2,048 MB | 1,839 MB ✅ | 2,214 MB ❌ |

Key differentiators:

1. **RAM compliance.** Qwen2.5 uses 1,839 MB — within budget with 209 MB headroom. SmolLM2 exceeds the budget by 166 MB, which is unacceptable on Thandiwe's 3 GB Mobicel Hero.

2. **Consistent output length.** Qwen2.5 reliably generates full 150-token responses. SmolLM2 frequently stops early, producing shorter explanations that may not provide the worked examples and step-by-step breakdowns learners need.

3. **Multilingual potential.** Qwen2.5 has strong multilingual capabilities, which supports the V1 English + Afrikaans language targets and the V2 African language expansion.

4. **Throughput.** At 19.0 tok/s, Qwen2.5 has the highest token throughput — even though SmolLM2 appears faster on wall-clock time, it generates far fewer tokens per response.

The RAM headroom concern (209 MB for embedding model + vector DB + app) is a known risk that P0-002 and P0-003 must validate. If the embedding model + vector DB exceed this budget, options include:
- More aggressive quantisation (Q3_K_S or IQ4_XS) of Qwen2.5 to reduce RAM
- Reducing n_ctx from 2048 to 1024 if prompts fit
- Falling back to SmolLM2 on 4 GB+ devices with a tiered strategy

---

# Consequences

### Positive

- Clear model target for all downstream work (P0-002 through P0-005, Phase 1 app)
- Empirically validated — decision backed by 144 benchmark runs (12 prompts × 3 runs × 4 models)
- 1,066 MB model file leaves room for embedding model (~50–100 MB) within the storage budget, though the 150 MB installed footprint budget is a separate concern for P0-003
- 19.0 tok/s throughput provides a responsive user experience
- llama.cpp runtime is well-maintained, has Android NDK support, and is permissively licensed (MIT)
- Qwen2.5's context window (32K native, running at 2K) provides flexibility to expand prompts if needed

### Negative / Trade-offs

- Tight RAM headroom (209 MB) — embedding model, vector DB, and Android app must fit in this margin
- Host-machine benchmarks may not perfectly reflect Android performance (mmap behaviour, OS memory accounting differ)
- Qwen2.5's generation time (8.0s) is close to the 10s limit — slower phones may breach it
- 1.5B parameter model has inherently limited reasoning depth compared to larger models — relies heavily on retrieval-grounded prompting (which aligns with ADR 0003)
- Model is from Alibaba (Qwen team) — need to verify license terms are compatible with project distribution

---

# Key Learnings

- Disk size ≠ RAM footprint. SmolLM2 (1,007 MB on disk) uses more RAM (2,214 MB) than Qwen2.5 (1,066 MB on disk → 1,839 MB RAM). Architecture and tokenizer design matter more than raw parameter count for memory efficiency.
- Short model outputs inflate throughput metrics. SmolLM2 appeared fastest but was generating incomplete responses. Always evaluate generation time for the target output length, not just tok/s.
- The 2 GB RAM budget is extremely tight — only 1 of 4 candidates fits. This is a hard physical constraint of the target hardware, not a negotiable threshold.
- Benchmarking on the host machine gives valid relative rankings but absolute numbers need Android validation.

---

# Agent Guidance

- **Model reference:** Qwen2.5-1.5B-Instruct, quantised as Q4_K_M GGUF, loaded via llama.cpp
- **RAM budget:** 1,839 MB for model inference — leaves ~209 MB for everything else in the 2 GB app budget
- **Context window:** Use n_ctx=2048. Do not exceed this in prompt construction. The model supports up to 32K natively but we constrain to 2K for RAM reasons.
- **Temperature:** 0.3 for curriculum content (low creativity, high factuality)
- **Max tokens:** 150 for standard explanations. May increase for worked examples if latency permits.
- **Prompt format:** Use the retrieval-grounded template from phone-architecture.md. The model follows instruction formatting well.
- **Fallback strategy:** SmolLM2-1.7B is the backup if RAM budget relaxes or if a Q3_K_S quantisation of Qwen2.5 proves insufficient quality
- **Do not** switch models without re-running the full benchmark suite and updating this ADR
- **Do not** increase n_ctx without measuring the RAM impact (each 512-token increase adds significant KV cache memory)
- **Validate on Android** — host benchmarks are directional only. P0-005 must confirm on a target emulator.
