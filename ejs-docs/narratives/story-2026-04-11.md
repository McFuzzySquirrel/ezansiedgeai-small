## Prologue

eZansiEdgeAI is an offline-first, privacy-first AI tutoring platform purpose-built for learners in underserved South African schools. The premise is audacious: every student with an Android phone should have access to a curriculum-aligned, step-by-step AI tutor — one that works without internet, without a school server, and without leaking a single byte of learner data.

Doug (McFuzzySquirrel) — the founding engineer and lead decision-maker — drove this project from a strategic vision document through to working, validated on-device AI inference. Throughout the journey, Doug collaborated with a rotating cast of GitHub Copilot agents: research agents, specialist sub-agents (project-orchestrator, ai-pipeline-engineer, android-ui-engineer, content-pack-engineer, and others), and the ejs-journey agent recording it all.

Three user personas grounded every decision: **Thandiwe**, a Grade 6 learner with a cracked-screen phone and no home WiFi; **Sipho**, her older brother helping her with fractions on a 2 GB RAM device; and **Ms. Dlamini**, their teacher who can pre-load content packs but can't guarantee a reliable school network.

This story covers 16 sessions across 40 days — from the first line of repository scaffolding on 3 March 2026 to the first live AI-generated tutoring response on a real learner device on 11 April 2026.

### Arc 1: The Vision Becomes Architecture
*(ejs-session-2026-03-03-01)*

#### What was happening

Everything starts with a vision document — `docs/vision.md` — a 717-line strategic description of what eZansiEdgeAI could be. Doug brought it to the first session with one instruction:

> *"look at the docs/vision.md, we need to start working on this, use as many sub agents as required and make sure to kick off the ejs system."*

No code existed. No repo structure. Just the vision.

#### How it unfolded

Copilot read the entire vision document first — all 717 lines — before writing a single file. From it, the architecture crystallized: a 4-layer Android app (App → AI → Data → Hardware), offline-first, phone-first, retrieval-augmented generation over local CAPS-aligned content packs.

Four parallel sub-agents went to work simultaneously. An **architecture sub-agent** produced four docs — `system-overview.md`, `phone-architecture.md`, `edge-device-architecture.md`, and `deployment-modes.md` — with ASCII diagrams, API tables, and capacity planning grounded in the vision's device specs. A **development sub-agent** produced `coding-principles.md`, `ai-agent-instructions.md`, and `backlog-v1.md` (24 work items spanning Phases 0–3). A **scaffolding sub-agent** built the directory skeleton — `apps/`, `models/`, `tools/` — with READMEs and a `models/.gitignore` to protect against accidentally committing multi-GB model files.

Then Doug issued a prompt that became a recurring pattern throughout the project: *"since this is new, we need ADRs for this."* Copilot applied the decision-detection rubric to the vision document and identified five decisions worthy of formal capture — each meeting the criteria of a system boundary change, a public contract definition, or a security/compliance posture shift. Five parallel **ADR sub-agents** produced ADRs 0001–0005 in a single pass.

Those five ADRs became the constitutional layer of the project:

- **ADR 0001** — *Phone-first*: the learner's phone is the primary compute surface; the school edge node is an optional accelerator, never a dependency.
- **ADR 0002** — *Offline-first*: zero network calls at runtime is a testable, enforceable constraint — not a fallback mode.
- **ADR 0003** — *Retrieval-first*: the model explains content retrieved from local curriculum packs; it does not invent answers from its own weights. The real product is the content pack, not the LLM.
- **ADR 0004** — *Edge as booster*: the school Raspberry Pi is an optional capability booster; learning must work fully without it.
- **ADR 0005** — *Privacy-first*: learner preferences captured but not performance; everything local, nothing transmitted.

#### What was decided

Five architectural decisions, formally captured, that would constrain every subsequent engineering choice. Agents were explicitly instructed: "Every feature must work on a phone with 3 GB RAM, Android 10+, no internet connection."

#### What was learned

A well-written vision document can drive concrete architecture decisions without additional input. The key technique: read the full document before writing anything, then decompose into independent parallel streams. Vision documents become most useful when grounded in real user personas — "48×48 dp touch targets" came from Thandiwe's cracked screen, not a generic accessibility guideline.

### Arc 2: Proving the Technical Bet
*(ejs-session-2026-03-03-02, ejs-session-2026-03-04-01, ejs-session-2026-03-11-01)*

#### What was happening

Phase 0 was the feasibility track — the question of whether the project was physically possible before building on top of it. Three spikes ran in sequence: LLM inference (P0-001), embedding + retrieval (P0-002 + P0-003 merged), and content pack + end-to-end pipeline (P0-004 + P0-005).

Doug named P0-001 "the single biggest technical risk in the project." If no model could fit within the 2 GB RAM budget and meet latency targets, the entire approach would pivot.

#### How it unfolded

**The LLM Spike (3 March):** Copilot selected four candidate models based on the phone architecture document's constraints: Qwen2.5-1.5B, SmolLM2-1.7B, Gemma-2-2B, and Phi-3.5-mini. All four were downloaded (~5.9 GB total), a Python benchmarking harness was built on a dedicated branch, and 4 × 12 × 3 = 144 evaluations ran. The result was decisive:

| Model | Disk | Load | Avg Gen | Peak RAM | Verdict |
|-------|------|------|---------|----------|---------|
| Qwen2.5-1.5B Q4_K_M | 1,066 MB | 0.8s | 8.0s | 1,839 MB | **PASS** |
| SmolLM2-1.7B Q4_K_M | 1,007 MB | 0.8s | 0.9s | 2,214 MB | FAIL (RAM) |
| Gemma-2-2B Q4_K_M | 1,629 MB | 1.2s | 12.5s | 2,900 MB | FAIL |
| Phi-3.5-mini Q4_K_M | 2,282 MB | 1.4s | 17.6s | 4,379 MB | FAIL |

Qwen2.5 was the only model that passed all three criteria. The critical insight — and the most counterintuitive learning of the whole project — was that disk size and RAM footprint are not correlated. SmolLM2 (1,007 MB on disk) consumed 2,214 MB of RAM. Qwen2.5 (1,066 MB on disk) used only 1,839 MB. Architecture, tokenizer design, and memory mapping behavior matter more than parameter count. Also notable: SmolLM2's speed advantage was partly illusory — it generated very short responses (far below `max_tokens=150`), inflating its tokens-per-second figures. Qwen2.5 consistently produced full 150-token responses.

ADR 0006 captured the decision: Qwen2.5-1.5B-Instruct Q4_K_M as V1 on-device LLM.

**The Embedding + Retrieval Spike (4 March):** The follow-on session tackled embedding model selection and vector store strategy — but it came with a significant scope change that Doug introduced. The minimum device floor was raised from 3 GB to 4 GB (marketed). This single constraint change unlocked enormous headroom: from 209 MB to 1,161 MB of available RAM overhead. The reason was grounded in market reality: 4 GB Android phones (Samsung A04s, Redmi 10C, Mobicel Hero) are mainstream in the South African low-end segment. Eliminating the 2 GB tier was simpler than building tiered architecture to accommodate it.

Three embedding models were benchmarked across multiple vector store options. `all-MiniLM-L6-v2` achieved 100% Top-3 retrieval accuracy. `bge-small-en-v1.5` exceeded the 100 MB embedding budget at 128 MB. `gte-small` passed accuracy but ran 3× slower. The decision was obvious. ADR 0007 captured it.

**The Content Pack + E2E Spike (11 March):** With models selected, the final feasibility session built the first content pack and wired the full pipeline together. The key format decision was SQLite — a single `.pack` file readable on Android without extraction, with per-row integrity checks and native API support. The full pipeline ran end-to-end: a CAPS-aligned Grade 6 fractions question flowed through content retrieval, prompt construction, and Qwen2.5 LLM inference in a single Python orchestration. ADR 0008 locked the format.

#### What was decided

- **ADR 0006**: Qwen2.5-1.5B-Instruct Q4_K_M as V1 on-device LLM (llama.cpp / GGUF)
- **ADR 0007**: all-MiniLM-L6-v2 + FAISS Flat as embedding + retrieval; device floor raised to 4 GB
- **ADR 0008**: SQLite as content pack format (`.pack` extension)

#### What was learned

Disk size does not predict RAM usage. Prefer simple constraint changes — raising the device floor — over complex architectural workarounds for multiple device tiers. SQLite is the right format for mobile curriculum delivery: no extraction step, native Android API, atomic transactions, per-row integrity. And benchmarking with the same runtime as production (llama.cpp) produces directly applicable results — don't benchmark in PyTorch if you're shipping llama.cpp.

### Arc 3: Assembling the Team
*(ejs-session-2026-03-14-01)*

#### What was happening

The PRD v1 existed. The feasibility spikes had validated the technical stack. But the PRD was 50 KB — spanning 4 phases, 24 work items, cross-cutting security and privacy requirements, and a 4-layer architecture. One agent working alone would be slow and error-prone. Doug asked: *"analyze prd and build out a team."*

#### How it unfolded

The `forge-team-builder` agent read the full PRD before designing anything. It mapped 10 functional requirement groups to specialist domains and proposed an 8-agent team with clear ownership boundaries:

- **project-architect** — Gradle modules, dependencies, build system, scaffold
- **android-ui-engineer** — All UI screens, Markdown and math rendering, accessibility
- **ai-pipeline-engineer** — embed → retrieve → prompt → generate pipeline
- **content-pack-engineer** — Pack format, loader, builder CLI, validator
- **learner-data-engineer** — Profiles, encryption, persistence, crash resilience
- **edge-node-engineer** — mDNS discovery, LAN sync, edge server (Phase 3 scope)
- **qa-test-engineer** — Full testing pyramid, security audit, acceptance criteria
- **project-orchestrator** — Coordination, sequencing, handoffs, execution tracking

The key design decision: security and privacy would be distributed as constraints across every agent, not isolated in a dedicated security role. A separate security agent would inevitably overlap with every other agent's domain. Instead, each agent owns security within its scope; `qa-test-engineer` owns verification.

Two reusable skills were identified: `scaffold-android-feature` and `create-content-chunk` — both high-frequency repeatable patterns that benefit from consistent structure.

#### What was decided

8-agent team with clear domain boundaries. Security as a cross-cutting constraint, not a separate agent. Two reusable skills packaged for recurring work.

#### What was learned

The PRD's 4-layer architecture maps naturally to agent boundaries when approached top-down. Related functional areas that seem separate in isolation (profiles + preferences + persistence) belong together when their data dependencies are traced. Read the full PRD before designing the team — the groupings become obvious.

### Arc 4: From Scaffold to Shipping App
*(ejs-session-2026-03-14-02, ejs-session-2026-03-14-03, ejs-session-2026-03-17-01)*

#### What was happening

Phase 1 — the Offline Learning Loop — was the first production implementation milestone. Starting from 9 Kotlin stubs and zero tests, the 8-agent team needed to deliver a working offline learning app with chat, topic browser, profile management, content pack loading, and secure data persistence.

#### How it unfolded

**14 March — Scaffold and deliberate pause:** The first Phase 1 session established the Android project scaffold — 9 Gradle modules, manual dependency injection, navigation, and Material 3 theme. Then Doug paused before proceeding to full implementation, a deliberate choice to confirm the scaffold was solid before building on it. A key discovery: Kotlin Compose compiler is now a Kotlin plugin (`kotlin.plugin.compose`) rather than a separate Compose compiler artifact, which simplified the build configuration considerably.

**17 March — The Big Build:** The largest single session of the project. Doug opened with two words directed at `project-orchestrator`: *"lets begin."* The orchestrator analyzed the PRD (50 KB), reviewed all 8 agent files, catalogued the existing project state (9 modules, 3 Kotlin stubs, zero tests), and produced an execution plan with 9 tasks and a dependency graph. Then it executed.

The implementation was a coordinated multi-agent operation across 7 specialist sub-agents:

**project-architect** delivered manual dependency injection via a single `AppContainer` class with Kotlin `lazy {}` delegates — ADR 0009. The rationale was transparency over convention: in a small team, explicit service wiring is more readable and auditable than code generation. All services are singletons; ViewModels receive dependencies via explicit factory implementations.

**learner-data-engineer** delivered AES-256-GCM field-level encryption with Android Keystore — ADR 0010 — plus WAL-mode SQLite for chat history and atomic file writes (temp → fsync → rename) for crash safety. The choice of `org.json.JSONObject` over `kotlinx.serialization` eliminated a dependency. Every learner-facing field — names, chat messages, preferences — is encrypted individually.

**content-pack-engineer** delivered double SHA-256 verification (before AND after copy to app storage), read-only SQLite flags, and a pure-Kotlin cosine similarity fallback for devices without ONNX Runtime support. The double-verification catches both download corruption and storage corruption.

**ai-pipeline-engineer** delivered the full RAG pipeline (embed → retrieve → prompt → generate), a Jinja2-style prompt template engine with a recursive-descent parser — ADR 0011 — and 98 unit tests for the template engine alone. The template engine supports `{{ }}` interpolation, `{% if/elif/else %}` conditionals, `{% for %}` loops, and `|` filters. Sequential model loading (embedding unloads before LLM loads) keeps peak RAM within budget.

**android-ui-engineer** delivered the chat screen with streaming responses, topic browser with breadcrumb navigation for CAPS tree traversal, profile selection, preferences, library, and onboarding tooltips — all in Jetpack Compose with LaTeX math rendering (no WebView). The `ChatWithQuestion` route enables topic browser → pre-filled chat navigation.

**qa-test-engineer** delivered 285 tests across all modules using hand-written fakes (no mocking framework — fakes preferred for clarity), a `FakeAndroidKeyStore` utility for unit-testable encryption, Robolectric for Android-dependent tests, and `testOptions.unitTests.isReturnDefaultValues=true`.

The outcome of coordinated execution: **84 files changed, 16,278 lines added** in a single session.

Then the build broke. Four compilation errors emerged when `assembleDebug` ran:
1. Missing `com.google.android.material` dependency for the Material theme
2. Duplicate companion objects in `PromptBuilder.kt`
3. Missing `activity-compose` for `BackHandler` in `feature:topics`
4. Missing launcher icon resources

Each was diagnosed and fixed iteratively across four build attempts. Build attempt four: **BUILD SUCCESSFUL**. Output: `ezansi-v0.1.0-debug.apk` — 128 MB, sideloadable to real phones.

Doug's responses were characteristic: *"lets add this to the readme."* Then: *"do we need to update the backlog."* All 8 Phase 1 tasks were marked complete with checked acceptance criteria, implementation notes, and completion dates.

#### What was decided

- **ADR 0009**: Manual DI via AppContainer — `lazy {}` singletons, ViewModel factories, zero reflection, zero code generation
- **ADR 0010**: AES-256-GCM with Android Keystore — field-level encryption for all learner data, `FakeAndroidKeyStore` for testing
- **ADR 0011**: Custom Jinja2-style template engine — recursive-descent parser in pure Kotlin, no external dependencies

#### What was learned

Sub-agents produce correct code for their own modules but consistently miss cross-module dependency wiring. The compilation step is the integration point — it catches everything the agents missed. Providing full file contents in agent prompts (not just references) consistently produces better results. Manual DI works well for small service graphs: the `AppContainer.kt` pattern is transparent, auditable, and testable with zero framework overhead.

### Arc 5: Getting to Real Devices
*(ejs-session-2026-03-29-01, ejs-session-2026-03-29-02)*

#### What was happening

The APK existed but AI components ran on mock implementations. Content packs were empty. No real model had ever loaded. Doug returned after a period away: *"ok, back again — I need to test this on the emulator, what do I need to do?"*

#### How it unfolded

The first challenge was Android Studio sync failures — `unresolved androidx.compose.ui:ui-test-junit4`. The `project-orchestrator` diagnosed a Gradle scope mismatch: Compose BOM was missing from `androidTestImplementation`. The fix was applied across all Compose feature modules. Build: successful.

Then came documentation. Doug asked for an emulator testing runbook for collaborators, then a sideload runbook, both linked from the README. The pattern of building first and documenting for the team immediately after was consistent throughout the project.

The content pack work began when Doug shared a CAPS curriculum PDF. The `project-orchestrator` translated it into 93 seed chunks covering all four Grade 6 Mathematics terms, built `maths-grade6-caps-all-terms-v1.0.pack`, and validated retrieval: **93.8% Top-3 accuracy — PASS**. One practical hitch: the local GPU was incompatible with the installed PyTorch build. The solution was CPU-only execution — a one-flag change with no architectural consequence.

The second session (29 March) opened the door to real runtime validation: a dedicated branch, runtime mode observability, and the first real device smoke test plan. The runbooks were updated with exact ADB commands for staging model files and content packs.

Doug also used this arc to polish the project's public blog post — through three editorial passes: *"lets update this post, keep my writing style"* → *"do one more editorial pass"* → *"do a sharper publication pass for blog rhythm and headline/subhead strength."* The result maintained his first-person, measured tone while gaining sharper headlines and tighter narrative rhythm.

#### What was decided

No new ADRs. The decisions were operational: CPU-over-GPU for build tooling (pragmatic), Compose BOM alignment across all modules (standard hygiene), ADB-based model staging workflow documented in runbooks for collaborators.

#### What was learned

The gap between "APK compiles" and "app runs with real content" is significant. Content packs, model files, and device configuration all need to be staged before meaningful testing can happen. Documentation runbooks should be built alongside the features they describe — not after. And GPU assumptions should always be made explicit: hardware compatibility varies across development machines.

### Arc 6: The Gap That Changed Everything
*(ejs-session-2026-04-10-01, ejs-session-2026-04-10-02)*

#### What was happening

Two problems had been accumulating in the codebase. First: Qwen2.5-1.5B on llama.cpp was producing 43+ second inference times on real devices — far slower than the 8-second host benchmark. Second, and more surprising: `OnnxEmbeddingModel.embed()` was never implemented. It always fell back to deterministic hash-based embedding. The semantic search capability the architecture promised had never actually worked.

Doug brought both observations to a research session: *"Research into Gemma 4 models and semantic search capability gaps."*

#### How it unfolded

The research agent explored the full codebase — AI pipeline code, ADRs 0006 and 0007, P0-001 and P0-002 spike reports, architecture docs, PRD constraints — and surfaced two key findings:

1. The RAG pipeline depended on pre-computed embeddings in content packs for retrieval quality. Runtime query embedding was hash-based — semantically random. Semantic search could not work until real embedding inference was integrated at query time.

2. Google's Gemma 4 1B model in MediaPipe LiteRT format could potentially replace *both* Qwen2.5-1.5B and all-MiniLM-L6-v2 with a single ~529 MB model. This would halve disk usage compared to the Qwen2.5 + MiniLM combination (~1,153 MB → ~529 MB), simplify the runtime to one SDK (eliminating llama.cpp's JNI complexity), and unlock MediaPipe's first-party GPU/NPU delegation.

The research agent attempted to fetch the Gemma 4 blog post directly but was blocked by the sandbox network. Working from published specifications and the codebase's own interface abstractions — `LlmEngine`, `EmbeddingModel`, `ContentRetriever` — the research document (`docs/research/gemma4-model-evaluation-and-semantic-search.md`) recommended a validation spike before committing engineering effort. The interface layer built in Phase 1 made the migration path architecturally clean.

The next session formalized this into a **Feature PRD**. The `forge-build-feature-prd` agent created a 422-line PRD covering five implementation phases (F1: validation spike, F2: engine integration, F3: pack re-embedding, F4: semantic search UI, F5: real-device validation). Then a **rubber-duck** validation agent reviewed the plan and surfaced three blocking gaps:
- The plan lacked a real-device validation split (automated tests ≠ device tests)
- Embedding parity between Android and Python tooling was not verified
- Parallel engine implementations were proposed before the model contract was defined

All three findings were adopted. A fourth suggestion — a feature flag system for toggling between engines — was set aside as unnecessary complexity for a binary migration.

Doug's role in this arc was as signal provider and quality gate: surfacing the performance gap, commissioning the research, evaluating the rubber-duck critique, and deciding what to build. The agents handled investigation and formalization.

#### What was decided

No ADRs from this arc — the research and PRD are input artifacts for the next arc's decision. The key pending decision: migrate to Gemma 4 UNIFIED (one model for both generation and embedding) vs. SPLIT (Gemma 4 for generation, retain MiniLM for embedding).

#### What was learned

The `OnnxEmbeddingModel` hash-based fallback had existed since Phase 1 without triggering any test failure — consistent hash-based embedding produces reproducible (if semantically meaningless) results that pass retrieval accuracy tests. Real embedding quality can only be validated by running actual semantic queries against a real model. External research should always be pre-fetched before starting, since sandboxed agents cannot reach the open web. The rubber-duck validation pass before large multi-agent executions consistently surfaces problems that pure planning misses.

### Arc 7: The Great Model Migration
*(ejs-session-2026-04-11-01, ejs-session-2026-04-11-02, ejs-session-2026-04-11-03)*

#### What was happening

The Feature PRD called for 33 implementation tasks across 5 phases, coordinated across 6 specialist agents. This was the largest and most complex coordinated multi-agent execution in the project. The outcome would migrate eZansiEdgeAI from Qwen2.5-1.5B (llama.cpp, GGUF) to Gemma 4 1B (MediaPipe GenAI SDK, LiteRT), replace the ONNX embedding model, re-embed all content packs to 768-dimensional vectors, add user-facing semantic search, and validate everything on a real learner device.

#### How it unfolded

**Phase F1 — Validation Spike (Decision Gate):** Before committing to the migration, `project-architect` added MediaPipe GenAI SDK 0.10.33 to the Gradle build as an additive dependency — keeping the legacy llama.cpp/ONNX stack intact until validation confirmed the migration path. The `ai-pipeline-engineer` built a benchmark harness with 17 tests. The decision gate evaluated two options:

- **UNIFIED**: Gemma 4 1B for both generation and embedding — single 529 MB model, shared provider, 768-dimensional embeddings
- **SPLIT**: Gemma 4 1B for generation only, all-MiniLM-L6-v2 retained for embedding

The gate selected **UNIFIED** — one model load saves approximately 800 MB RAM and eliminates a second SDK dependency. ADR 0012 captured the decision: Gemma 4 1B as the unified on-device model, superseding both ADR 0006 (Qwen2.5-1.5B) and ADR 0007 (all-MiniLM embedding model).

**Phase F2 — Engine Integration:** The `ai-pipeline-engineer` delivered six components across six commits: `GemmaModelConfig`, `GemmaModelProvider`, `GemmaLiteRtEngine`, `GemmaEmbeddingModel`, `PromptBuilder` with `GEMMA_TURN` format, and unified model loading in `AppContainer`. The critical architectural choice: a single `GemmaModelProvider` is shared between LLM inference and embedding — one model instance serves both roles, staying within RAM budget. The legacy stack was retained behind a fallback path. The `qa-test-engineer` added 30 integration tests covering the unified pipeline.

**Phase F3 — Content Pack Re-embedding:** The `content-pack-engineer` produced `EMBEDDING_CONTRACT.md` — a canonical specification for pack embedding schemas — and updated `build_pack.py` and `validate_pack.py` to produce Schema v2 packs with 768-dimensional embeddings (matching Gemma 4's output dimension). A `PackVersionDetector` with 22 tests handled backward compatibility detection at query time rather than installation time — users see their packs even if incompatible, with a clear upgrade message. Both existing content packs were re-embedded.

**Phase F4 — Semantic Search:** The `ContentSearchEngine` API was built and wired into `AppContainer`. The android-ui-engineer delivered `SearchUiState`, `SearchViewModel`, `SearchResultCard`, and `SearchResultsList` with a `hasSearched` flag to correctly distinguish "no results found" from "hasn't searched yet" — a subtle UX distinction with big impact on the empty state experience. Material 3 search bar integration and Ask AI navigation followed.

**Phase F5 — Real-Device Validation. The moment the project had been building toward.**

Doug pushed `gemma4-1b.task` (529 MB) to a vivo V2434 device via ADB. The app launched. The model began loading. Then:

```
NoSuchMethodException: setTemperature [float]
```

The `project-orchestrator` investigated by running `javap` on the MediaPipe GenAI AAR — enumerating the actual SDK methods rather than relying on documentation. The finding: in SDK version 0.10.33, `setTemperature`, `setTopK`, and `setRandomSeed` had been moved from `LlmInferenceOptions.Builder` (model-level configuration) to `LlmInferenceSessionOptions.Builder` (session-level configuration). The code had been written against an earlier API surface.

The fix: remove session-level parameters from `createLlmInference()`, move them to session config, add `setPreferredBackend(Backend.GPU)` via reflection.

Fix applied. First test run: **native crash during GPU inference.** The agent changed `Backend.GPU` to `Backend.DEFAULT`, letting the SDK auto-detect the best available backend. On the vivo V2434, that meant CPU inference via XNNPack neural network computation.

Second test run:

Model loaded in **7.4 seconds**. Pipeline completed in **43 seconds**. The response appeared on screen:

> *"Okay! Let's learn about fractions. They are just parts! Like sharing cookies or building something!"*

**eZansiEdgeAI's first successful on-device AI-generated tutoring response on a real learner device.**

Three more bugs emerged in the manual testing walkthrough that followed:

- **Topics crash**: `IllegalArgumentException: Key "term1" already used` — both content packs shared the same root tree node IDs in their CAPS topic structure. `LazyColumn` requires unique keys within a list. Fixed by implementing `mergeTopicTrees()` in `TopicsViewModel`, deduplicating by path rather than dropping content.
- **Search unavailable**: `GemmaEmbeddingModel.isLoaded()` was gated on `GemmaModelProvider.isLoaded()` — but hash-based embedding doesn't need the Gemma model to be loaded. Search was blocked until after the first AI query. Fixed: `isLoaded()` returns `true` unconditionally for hash-based implementation.
- **Navigation confusion after Ask AI**: Bottom bar route matching used exact string equality. When routing to `chat?question=What are fractions?`, the `chat` route didn't match. Fixed to use `startsWith()` for route comparison, correctly highlighting the chat tab for parametric routes.

All fixes committed. `DEVICE-VALIDATION-CHECKLIST.md` updated from Conditional Pass to: **PASS ✅**

Final test session results: direct chat, 3 rapid consecutive searches, Ask AI → chat flow — all passed with no crashes or freezes. Pipeline: 46–55 seconds warm. Response length: 605 characters for "What are fractions?"

#### What was decided

- **ADR 0012**: Gemma 4 1B (MediaPipe GenAI SDK) as unified on-device model — supersedes ADR 0006 and ADR 0007
- Use `Backend.DEFAULT` rather than `Backend.GPU` — let the SDK auto-detect; avoids native crashes on devices without adequate GPU compute
- Merge topic trees by path for multi-pack deduplication — `LazyColumn` requires unique keys
- `isLoaded()` on hash-based implementations should always return `true` — functional readiness, not model state
- Use `startsWith()` for bottom bar route matching — parametric routes break exact equality

#### What was learned

Inspect SDK APIs with `javap` rather than relying on documentation — API surfaces change between minor versions without always being reflected in changelogs or online references. GPU delegation causes native crashes on CPU-only or limited-GPU devices; `Backend.DEFAULT` is the safe default for production distribution. Multi-agent parallel execution works cleanly when file boundaries don't overlap; conflicts happen at integration seams, not within domains. And the most important structural insight: the `LlmEngine` / `EmbeddingModel` / `ContentRetriever` interface layer built correctly in Phase 1 made the Gemma 4 migration require no architectural change — only new implementations behind existing interfaces. Abstractions that anticipate future model swaps are worth the upfront design investment.

### Arc 8: Shipping Clean
*(ejs-session-2026-04-11-04)*

#### What was happening

Before pushing to GitHub, Doug noticed something: *"before we commit and push again — need to make sure we are not pushing the model files, rather instruct users how to build and push to device."*

#### How it unfolded

`git ls-files models/` revealed that `gemma4-1b.task` (529 MB) was tracked by git. The existing `models/.gitignore` covered `*.gguf`, `*.onnx`, `*.bin`, `*.pt`, and `*.safetensors` — but `*.task` was absent. The MediaPipe format had been introduced during the Gemma 4 migration without a corresponding gitignore update. A format gap caused by adding a new model type without auditing the ignore rules.

Copilot added `*.task` to `models/.gitignore`, ran `git rm --cached models/phone-models/gemma4-1b.task` to untrack without deleting the local file, and rewrote `models/phone-models/README.md` to give contributors clear instructions: Kaggle download steps for the Gemma 4 model, followed by exact `adb push` commands for staging to a device.

The first push attempt was rejected by GitHub with a clear error: 529 MB exceeds the 100 MB file size limit. The binary existed in the history of commit `c5109d7` — even though HEAD had already removed it. `git rm --cached` removes a file from future tracking but doesn't rewrite history.

`git filter-branch` rewrote the three unpushed commits to strip `gemma4-1b.task` from the index at the historical commit where it was added. Second push attempt: **50 MB uploaded** (versus 424 MB on the first attempt). GitHub accepted it.

The PROGRESS.md was updated to show the storage footprint of the active model set: 529 MB (Gemma 4) + 87 MB (embedding) = **616 MB total** — well within the 1.4 GB first-launch budget.

#### What was decided

`*.task` added to `models/.gitignore`. Three unpushed commits rewritten via `git filter-branch` to remove the binary from history. Future contributors directed to Kaggle for model download and ADB for device deployment.

#### What was learned

Always add new model file extensions to the gitignore in the same commit that introduces that model format — format gaps cause binaries to slip through at every format transition. `git ls-files <directory>` is the definitive pre-push check — `git status` alone does not reveal files that were committed in earlier commits. `git rm --cached` is the correct tool for untracking without deletion; `git filter-branch` (or `git-filter-repo`) is the correct tool when the binary is already in reachable history.

## Epilogue

On 11 April 2026, eZansiEdgeAI delivered its first real AI-generated tutoring response on a real learner device. The journey from vision document to working app took 40 days and 16 sessions.

**Current state of the system:**

- **Gemma 4 1B** (MediaPipe GenAI SDK 0.10.33) powers both on-device generation and embedding via a unified model provider
- **SQLite content packs** (Schema v2, 768-dimensional embeddings) cover Grade 6 Mathematics, CAPS-aligned, all four terms
- **Semantic search** is integrated into the UI with Material 3 design and Ask AI navigation flow
- **Encryption**: AES-256-GCM with Android Keystore — all learner data is field-level encrypted, nothing transmitted
- **Test suite**: 367 passing tests (18 pre-existing Robolectric failures unrelated to Gemma 4 work)
- **Real-device validated**: PASS on vivo V2434 (Android 14, Funtouch OS 14, CPU inference via XNNPack)
- **Repo hygiene**: All model binaries excluded from git; contributor workflow documented in `models/phone-models/README.md`

**Open questions and next frontiers:**

- *Real semantic embedding*: `GemmaEmbeddingModel` currently uses hash-based embedding. Extracting real embeddings from Gemma 4's final hidden layer is the next technical milestone for true semantic search.
- *Inference performance*: 43-second cold inference on the vivo V2434 is functional but not conversational. Warm inference is 46–55 seconds. GPU delegation on capable devices could reduce this significantly — unexplored.
- *Device testing breadth*: TalkBack accessibility, preferences sync, metric system compatibility — planned in the F5.7 test plan but not yet executed.
- *The edge node* (Phase 3): School Raspberry Pi content distribution exists only in ADR 0004 and `apps/school-edge-node/`. Phase 3 has not begun.

**Collaboration pattern observed across 40 days:**

The project ran a consistent **spike → benchmark → ADR → build → validate** loop. Every major technical decision was grounded in empirical evidence before implementation began. The multi-agent model worked cleanly when boundaries were file-level or module-level, and broke down at integration seams — cross-module dependencies, shared interfaces, and navigation wiring required human review of the compilation output to catch.

The rubber-duck validation pass before large multi-agent executions consistently surfaced problems that pure planning missed. The ejs-journey recording system captured decisions at the moment they occurred, not reconstructed afterward — which made this story possible to write from first principles.

What started as a vision document — one that insisted the phone is the platform, offline is the default, and privacy is non-negotiable — had its first real conversation with a learner device.

The app said: *"Okay! Let's learn about fractions. They are just parts!"*
